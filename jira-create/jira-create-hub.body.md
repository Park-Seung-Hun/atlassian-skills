## Step 0 — 공통 설정

> **[필수] 실행 순서 정책**
> - 본 스킬을 호출하면 **반드시 이 `## Step 0`부터 순차 실행**한다.
> - 사용자 입력에 SDD 파일 경로·프로젝트 키·요약 등 작업 대상 정보가 포함되어 있어도 **Step 0를 건너뛰지 않는다**. 작업 대상 정보는 Step 1 이후에서 소비한다.
> - 처리 순서: `0-0 config 로드` → (로드 성공 시) `0-0b 현재 설정 확인` → 사용자 응답에 따라 `0-0c 항목별 수정`(루프) 또는 곧바로 본문 `## Step 1` 진입. config 로드 실패 시엔 `0-1` 전체 fallback 후 `## Step 1`.
> - `$ARGUMENTS`에 **config의 PROJECT_KEY와 다른** 프로젝트 키가 포함된 경우만 예외: `0-2` 경로로 진입하며 `0-0b/0-0c`를 스킵한다. **같은 프로젝트 키**가 포함된 경우(예: config PROJECT_KEY=JST인데 사용자가 "JST에"로 재언급한 경우)는 오버라이드가 아니므로 일반 경로로 0-0b부터 진행한다 (0-2 섹션 진입 조건 참조).
> - **Step 1 진입 조건**: `0-0b` 선택 완료 또는 `0-1` fallback 완료 또는 `0-2` 재탐색 완료.
> - **0-0a 자동 customfield 키 검증**은 Step 0 진입 단계에서 실행하지 않는다. 본문 스킬이 **Jira 생성 직전**(배치: Step 4 "이대로 생성" 확정 직후, 단건: Step 5 확인 직후)에 명시적으로 호출하는 **지연 실행 서브루틴**이다.

> **실행 환경 전제**
> - `mcp-atlassian`, `notion`, `slack` MCP 서버가 등록돼 있어야 한다(스킬에 따라 사용하는 서버는 일부만 해당).
> - 본문에서 `mcp__<server>__<tool>` 풀네임으로 표기된 도구는 Claude Code/Codex 양 환경에서 동일하게 모델에 노출된다.
> - 본문에 "AskUserQuestion"이라고 적힌 부분은 구조화 질문 도구를 의미한다. 해당 도구가 없는 환경(예: Codex)에서는 동일 의미의 자연어 질문으로 대체하되, 선택지/검증 조건은 본문에 명시된 그대로 유지한다.
> - 설정 파일 경로 `{{CONFIG_PATH}}`는 두 환경이 워크스페이스 루트 기준으로 공유한다.

### 커스텀 필드 설정

커스텀 필드 ID는 0-0(config 로드) 또는 0-1(config 없을 때의 인라인 수집)에서 확정된다.
이하 지시에서 `{FIELD_SP}` / `{FIELD_AC}` / `{FIELD_EV}`는 확정된 필드 키를 의미한다.
config 로드 경로인 경우 본문 스킬이 Jira 생성 직전에 `0-0a` 서브루틴을 호출하여 customfield 키의 존재 여부를 1회 검증한다.
`(none)`으로 설정된 슬롯은 해당 수집 Step 전체를 스킵한다.

> **중요**: `description` 필드에 AC/증거를 포함하면 Jira가 자동으로 파싱하여 섹션이 잘리거나 커스텀 필드에 잘못 매핑된다.
> description = 목적 + 범위만 작성하고, AC와 증거는 반드시 커스텀 필드로 분리하여 전달한다.

### 0-0. config 로드

`{{CONFIG_PATH}}` 파일을 읽어 아래 값을 로드한다.

- **PROJECT_KEY**: `프로젝트 키` 항목
- **BOARD_ID**: `보드 ID` 항목
- **FIELD_SP**: `스토리 포인트 필드` 항목 (`(none)`이면 스토리 포인트 수집 Step 스킵)
- **FIELD_AC**: `AC 필드` 항목 (`(none)`이면 AC 수집 Step 스킵)
- **FIELD_EV**: `증거 필드` 항목 (`(none)`이면 증거 수집 Step 스킵)
- **DEFAULT_PRIORITY**: `기본 우선순위` 항목 (없거나 `(none)`이면 스킬별 fallback 사용)
- **DEFAULT_EVIDENCE**: `기본 증거 형태` 항목 (없거나 `(none)`이면 스킬별 fallback 사용)
- **SLACK_ID**: `Slack 사용자 ID` 항목 (`(none)`이면 Slack 알림 Step 스킵)

로드 후 **PROJECT_ID 확보**(세션 캐시용) — 아래 폴백 체인을 **첫 성공까지 순차 시도**한다. 0-0a / 0-1 슬롯별 서브루틴이 scope 필터용으로 재사용한다.

1. `jira_search({"jql":"project = {PROJECT_KEY}","fields":"project","limit":1})` 호출 → 응답 `issues[0].project.id` 추출. MCP 응답 스키마에 따라 `project` 객체에 `id`가 누락될 수 있음 — 그 경우 2단계로.
2. `jira_get_all_projects()` 호출 → 응답 배열에서 `key == {PROJECT_KEY}` 인 엔트리의 `id` 추출.
3. `jira_get_issue({issue_key: 기존 JST 이슈 키, fields: "*all"})` 응답에서 **PROJECT 스코프 customfield들의 `scope.project.id`가 단일 값으로 일관**되면 그 값을 PROJECT_ID로 간주(휴리스틱 폴백). 기존 이슈가 전혀 없거나 일관되지 않으면 실패 처리.
4. 위 모두 실패 → PROJECT_ID = `unknown`. 이후 scope 필터는 "**글로벌 필드만 통과** + 기타 전부 제외" 규칙으로 보수 적용한다 (아래 0-1 공통 프리앰블 참조).

확보 시도는 **최대 3회**. 성공 즉시 캐시하며, 이후 세션에서는 재조회하지 않는다.

### 0-0b. 현재 설정 확인

> **[차단 지점]** 이 절은 Step 1로 진입하기 전 **반드시 사용자 응답을 받고 종료**한다. config 로드가 성공한 경우 요약 출력과 선택지 질문을 생략할 수 없다. "바로 본문을 실행하는 게 사용자 의도로 보인다"는 판단으로 이 절을 건너뛰는 것을 금지한다.

**실행 조건**: 0-0에서 config 로드가 성공하고, 0-2 $ARGUMENTS 오버라이드가 발동되지 않은 경우에만 실행. config 파일이 없거나 PROJECT_KEY가 `YOUR_`로 시작하면 0-0b를 건너뛰고 0-1로 진입. 0-2 경로도 0-0b/0-0c/0-0a 전체를 건너뜀.

로드된 config 값을 아래 형식으로 요약 출력한다.

```
현재 설정 ({{CONFIG_PATH}}):
- 프로젝트 키       : {PROJECT_KEY}
- 보드 ID           : {BOARD_ID}
- 스토리 포인트 필드: {FIELD_SP}
- AC 필드           : {FIELD_AC}
- 증거 필드         : {FIELD_EV}
- 기본 우선순위     : {DEFAULT_PRIORITY}
- 기본 증거 형태    : {DEFAULT_EVIDENCE}
- Slack 사용자 ID   : {SLACK_ID}
```

AskUserQuestion (단일 선택):
> "위 설정으로 진행할까요, 항목을 수정할까요?"
- **그대로 진행** → 본문 `## Step 1` 진입 (customfield probe는 생성 직전에 수행)
- **항목 수정** → 0-0c 진입

### 0-0c. 항목별 수정

AskUserQuestion (**복수 선택**):
> "수정할 항목을 선택하세요."

선택지:
- 프로젝트 키 / 보드 ID (함께 수정) → **0-1-PROJECT** 재실행
- 스토리 포인트 필드 (FIELD_SP) → **0-1-SP** 재실행
- AC 필드 (FIELD_AC) → **0-1-AC** 재실행
- 증거 필드 (FIELD_EV) → **0-1-EV** 재실행
- 기본 우선순위 → 직접 입력 (예: Highest, High, Medium, Low, Lowest 중 하나)
- 기본 증거 형태 → 직접 입력 (예: 링크, 스크린샷, 로그, 테스트 결과 등 자유 텍스트)
- Slack 사용자 ID → 직접 입력 (Slack member ID, 예: U01ABCDEFGH). "(none)" 입력 시 알림 비활성화.

선택된 항목을 순차적으로 수집한 뒤 `{{CONFIG_PATH}}` 갱신.

**config 갱신 로직**:
1. `{{CONFIG_PATH}}`를 Read로 로드 (전체 내용).
2. 저장 포맷은 `라벨: 값` 라인 단위. 각 항목의 라벨은 아래와 같이 고정(이 라벨이 바뀌면 `jira-create-setup`과 호환성이 깨진다):
   - `프로젝트 키:`, `보드 ID:`, `스토리 포인트 필드:`, `AC 필드:`, `증거 필드:`, `기본 우선순위:`, `기본 증거 형태:`, `Slack 사용자 ID:`
3. 수정된 슬롯에 해당하는 **라인만** 새 값으로 교체. 다른 라인과 파일 내 다른 섹션은 건드리지 않는다.
4. Write로 전체 파일을 덮어쓴다.
5. "`{{CONFIG_PATH}}`를 갱신했습니다." 안내 출력.

완료 후 **0-0b로 복귀**하여 갱신된 값을 재확인한다 (사용자가 다시 "수정"을 선택하면 추가 수정 가능).

### 0-0a. 자동 customfield 키 검증 (지연 실행 서브루틴)

> **[중요] 호출 규약**
> - 이 서브루틴은 Step 0 진입 단계에서 **자동 실행되지 않는다**. 본문 스킬이 Jira 생성 직전에 명시적으로 호출한다 (배치: Step 4 "이대로 생성" 확정 직후 · 단건: Step 5 확인 직후, Step 6 첫 호출 이전).
> - 호출 조건: config 로드 경로(0-0 성공)로 진입한 세션에 한함. `0-1` fallback 완료 세션과 `0-2` 오버라이드 세션은 호출하지 않는다 (수집 중 scope 필터로 이미 검증됨 / 1회성 오버라이드).
> - **입력**: 0-0에서 확정·캐시된 `PROJECT_KEY`, `PROJECT_ID`, `FIELD_SP` / `FIELD_AC` / `FIELD_EV` 값.
> - **출력**: 아래 3가지 중 하나.
>   - `통과` — 모든 슬롯 검증 통과. config/세션 변경 없음.
>   - `재지정` — 실패 슬롯을 재지정·비활성화 경로로 처리 완료. config 또는 세션 필드 맵이 변경됨. 호출 측 본문은 **생성 전 반드시 사용자 재확인**을 수행한다 (아래 "재지정 반환 후 호출 측 재확인" 참조).
>   - `중단` — 사용자가 중단 선택. 호출 측 본문은 생성을 수행하지 않고 스킬 종료.
> - **세션 내 멱등**: 한 세션에서 `통과`로 반환된 뒤에는 본문 스킬이 재호출하더라도 실제 probe(`jira_search_fields`)를 재실행하지 않는다. 세션 플래그 `customfield_probe_passed = true`로 기록하고 즉시 `통과` 반환. 재지정으로 config가 갱신된 경우는 플래그가 해제(false)된 상태에서 새로 probe한다.

로드된 customfield 키가 현재 Jira 인스턴스에서 실제 조회 가능한지 1회 검증한다.

**probe**:
- `jira_search_fields`를 **`keyword` 인자 없이(또는 빈 문자열)** `limit: 200`으로 **정확히 1회**만 호출하여 customfield 전체 목록을 얻는다.
  - 이 호출을 customfield 키별로 분할하거나(`keyword="customfield_XXXXX"`), 카테고리별로 나누거나(`keyword="acceptance criteria"`, `keyword="evidence"` 등), 응답이 잘려 보인다고 재시도하는 것을 **금지**한다.
  - 응답이 길어 절단된 것으로 보여도 판정에 필요한 건 `id` 일치와 `scope` 확인뿐이며, `limit: 200`이면 통상 Jira 인스턴스의 customfield 전량을 포함한다. 검증용 호출에서 추가 탐색·교차 확인을 수행하지 않는다.
- `(none)`으로 설정된 슬롯은 검증 대상에서 제외한다. 모든 슬롯이 `(none)`이면 probe 자체를 수행하지 않고 즉시 `통과` 반환.
- config의 FIELD_SP / FIELD_AC / FIELD_EV 각 값에 대해 아래 순서로 판정한다:
  1. `id` 필드가 일치하는 항목을 찾는다. 없으면 **실패 (미존재)**.
  2. 해당 항목의 `scope` 필드를 확인:
     - `scope` 키가 **없음** → 글로벌 필드. **통과**.
     - `scope.type == "PROJECT"` 이고 `scope.project.id`가 0-0에서 캐시한 **PROJECT_ID와 같음** → 현재 프로젝트 전용 필드. **통과**.
     - `scope.type == "PROJECT"` 이고 `scope.project.id`가 PROJECT_ID와 **다름** → **실패 (다른 프로젝트 전용)**.
     - PROJECT_ID가 `unknown`인 경우(빈 프로젝트) → `scope` 없는 필드만 통과, PROJECT 스코프 필드는 일괄 실패 처리.

> 이 검증은 "필드 존재 + 현재 프로젝트 스코프 바인딩"까지 함께 확인한다. 동일한 scope 필터를 0-1 슬롯별 후보 제시에도 적용한다.

**실패 감지 시 분기 (슬롯별 순차 처리)**:

어떤 슬롯이 위 판정에서 실패하면 해당 슬롯에 대해 아래를 수행한다. 실패 사유에 따라 안내 문구를 달리한다. **안내 문구에는 실패 사실만 담고, 대체 후보값은 절대 함께 노출하지 않는다** (후보는 0-1-{슬롯} 재진입 UI에서만 제시).

```
# 미존재 — 안내 예시 (이 문구 외 정보 추가 금지)
FIELD_AC(`customfield_99999`)가 jira_search_fields 결과에서 확인되지 않습니다.
```

```
# 다른 프로젝트 전용 — 안내 예시 (이 문구 외 정보 추가 금지)
FIELD_AC(`customfield_XXXXX`)는 현재 프로젝트 `{PROJECT_KEY}`에서 사용할 수 없습니다(다른 프로젝트 전용 필드).
```

AskUserQuestion (단일 선택):
> "어떻게 처리할까요?"
- **재지정** → 해당 슬롯의 0-1 서브섹션(`0-1-AC` 등) 재진입 → 새 값 확정 → 0-0c의 config 갱신 로직으로 `{{CONFIG_PATH}}` 저장 → 다음 실패 슬롯으로 이동.
- **비활성화** → 세션 내에서만 해당 슬롯을 `(none)`으로 처리. `{{CONFIG_PATH}}`는 **변경하지 않는다**(의도적). 다음 실행 시 동일 오류가 재발할 수 있음을 안내.
- **중단** → 스킬 즉시 종료. 서브루틴은 `중단`을 반환하고 호출 측 본문은 생성을 수행하지 않는다.

> **[필수] 재지정 경로에서 지켜야 할 3가지 (어느 하나라도 빠뜨리지 말 것)**
>
> **(1) 0-1 서브섹션 재진입 강제**
> 재지정 선택 직후 **0-1-SP / 0-1-AC / 0-1-EV 중 해당 슬롯 서브섹션을 반드시 호출**한다. "현재 값: {customfield_xxxxx}. 재수집하시겠습니까? [예/아니오]" 질문을 먼저 표시하고(이 질문은 0-0c 재진입뿐 아니라 **0-0a 재지정 진입 시에도 동일하게 표시**한다), "아니오"면 해당 슬롯 비활성화 또는 다음 슬롯으로 진행. "예"면 키워드 매칭 → scope 필터 → 후보 제시 → "**매칭 1개인 경우에도 확인 질문 → 예/아니오 응답 수신**" 또는 "**매칭 0개/2개 이상인 경우 후보 선택 UI**"를 반드시 거친 뒤에만 새 값을 확정한다.
>
> **(2) config 저장 도구 제한**
> 확정된 값을 `{{CONFIG_PATH}}`에 반영할 때 사용 가능한 도구는 **`Read`와 `Write` 단 두 가지뿐**이다. 아래 도구·방식을 사용한 config 치환을 **전부 금지**한다:
> - `Bash` 도구를 통한 `sed`, `perl`, `awk`, `gsed`, `python -c`, `ruby -pi`, `cat > ...`, `echo "..." > ...`, `tee`, 리다이렉션(`>`/`>>`) 등.
> - `Edit` 도구의 in-place 부분 치환.
>
> 저장 절차는 정확히: `Read`로 전체 파일 로드 → 메모리에서 해당 라인만 교체 → `Write`로 전체 파일 덮어쓰기. 이 순서·도구 조합을 벗어난 저장은 수행하지 않는다. `Write` 실패 시 `Bash` 치환 도구로 우회하지 말고 사용자에게 저장 실패 사실을 알리고 중단한다.
>
> **(3) 실패 안내에 후보값 동반 노출 금지**
> "미존재" / "다른 프로젝트 전용" 안내 문구에는 **실패 사실만** 담는다. "현재 JST에서 쓰이는 AC 후보는 …" 같은 대체 후보를 같이 출력하거나, 실패 안내와 재지정 UI를 같은 턴에 합쳐 "후보 X로 바꿀까요?" 류의 단일 질문으로 축약하는 것을 금지. 대체 후보는 반드시 0-1-{슬롯} 재진입 UI에서만 제시한다.

모든 실패 슬롯 처리가 끝나거나 검증이 바로 통과하면 `customfield_probe_passed = true`를 세팅하고 **호출 측 본문으로 복귀**한다 (Step 1로 점프하지 않는다 — 호출 지점에 따라 Step 5 또는 Step 6으로 진행). 이때 config/세션 필드 맵이 변경됐다면 `재지정`을, 변경이 없었다면 `통과`를 반환 값으로 넘긴다.

> **[필수] 재지정 반환 후 호출 측 재확인**
> 서브루틴이 `재지정`을 반환한 경우, 호출 측 본문은 **즉시 Jira 생성 호출을 수행하지 않는다**. 아래 질문을 먼저 출력하고 응답을 받은 뒤에만 다음 단계로 진입한다.
> - AskUserQuestion(단일 선택): "설정이 변경됐습니다 ({변경 슬롯 요약, 예: `AC 필드: customfield_99999 → customfield_12881`}). 이 설정으로 생성을 이어갈까요?"
>   - **이어서 생성** → Step 5(배치) 또는 Step 6(단건) 진행.
>   - **취소** → 생성 없이 종료.
> 이전 (C) "생성" 응답을 근거로 사용자 재확인을 생략하고 바로 `jira_create_issue` / `jira_batch_create_issues` 등을 호출하는 것을 금지한다. config 상태가 달라졌으므로 사용자 동의를 새로 받아야 한다.

### 0-1. 인라인 수집 서브루틴

아래 네 개 서브섹션은 독립적으로 호출 가능하다. config 미설정 fallback에서는 0-1-PROJECT → 0-1-SP → 0-1-AC → 0-1-EV 순차 호출 후 반드시 **0-1-SAVE**로 `{{CONFIG_PATH}}`를 신규 작성한다. 0-0c/0-0a 재지정 분기는 해당 슬롯 서브섹션만 호출하며, 저장은 0-0c의 갱신 로직이 담당한다.

> **[필수] 사용자 확인 강제**: 각 0-1 서브섹션은 **사용자에게 질문하고 응답을 수신한 뒤에만** 다음 슬롯으로 진행한다.
> - 사용자 프롬프트에 프로젝트 키·필드 값이 이미 포함되어 있어도 슬롯별 확인 질문을 건너뛰지 않는다. "사용자가 이미 값을 줬다"는 판단으로 자동 확정하는 것을 금지한다.
> - 기존 이슈를 `jira_get_issue(fields="*all")` 등으로 조회해 customfield를 **역추정한 결과를 사용자 확인 없이 바로 채택하는 행위를 금지**한다. 역추정 자체는 허용되나 결과는 반드시 "매칭 1개" 분기의 확인 질문 또는 "매칭 0개/2개 이상" 분기의 선택 UI를 거쳐 사용자가 명시적으로 확정해야 한다.
> - "매칭 1개"인 경우에도 `"스토리 포인트 필드로 {필드명} ({필드 키})을 사용하시겠습니까?"` 형태의 확인 질문을 **반드시 출력**하고 예/아니오 응답을 받은 뒤에만 확정한다.

> **인라인 모드 SLACK_ID 처리**: config 미설정 인라인 fallback에서는 `SLACK_ID = (none)`으로 고정한다. Slack 알림은 `/jira-create-setup`을 통해서만 활성화된다.

> **[필수] customfield 후보 scope 필터 (0-1-SP / 0-1-AC / 0-1-EV 공통)**: `jira_search_fields` 결과에서 키워드 매칭으로 수집한 customfield 후보를 사용자에게 **제시하기 전에** 아래 scope 필터를 반드시 적용한다. 다른 프로젝트 전용 필드를 후보에 노출하는 것을 금지한다.
>
> 1. 0-0에서 캐시한 **PROJECT_ID**를 가져온다. 캐시가 없으면 0-0의 PROJECT_ID 확보 폴백 체인(1~4단계)을 즉시 실행하여 확보를 다시 시도한다.
> 2. PROJECT_ID 확보 결과에 따라 필터링:
>    - **PROJECT_ID 확보 성공**: 각 customfield 후보의 `scope`를 확인
>      - `scope` 키 **없음** → 글로벌 필드. **후보 유지**.
>      - `scope.type == "PROJECT"` 이고 `scope.project.id == PROJECT_ID` → 현재 프로젝트 전용 필드. **후보 유지**.
>      - 그 외 → **후보에서 제외** (로그에 "다른 프로젝트 전용 필드 제외: `{key}` (scope.project.id=`{id}`)" 1줄 남기고 사용자 선택지에는 **절대 노출하지 않음**).
>    - **PROJECT_ID = unknown** (폴백 체인 모두 실패): 보수적으로 처리
>      - `scope` 키 **없음** → 글로벌 필드. **후보 유지**.
>      - `scope.type == "PROJECT"` → PROJECT_ID 매칭 확인 불가. **후보에서 전원 제외**한다 ("현재 PROJECT_KEY와 매칭 확인 불가한 project-scoped 후보를 보수 제외" 안내 1줄 출력). 이 상태에서 다른 프로젝트 소유 후보를 사용자에게 떠넘겨 고르게 하는 것을 금지한다.
> 3. 필터링 후 후보가 0개면 사용자에게 아래 안내를 출력하고 `사용 안 함` / `직접 입력`만 제시한다:
>    > "키워드 매칭은 됐으나 현재 프로젝트 `{PROJECT_KEY}`에 사용 가능한 customfield를 자동 식별하지 못했습니다. 필드 key를 직접 입력하거나 사용 안 함을 선택하세요."
> 4. 0-0a에서 "재지정"으로 진입한 경우에도 같은 필터를 적용해, 재지정 후보에 다른 프로젝트 전용 필드가 올라오는 것을 방지한다.
> 5. 글로벌 필드(`scope` 없음)는 **어느 프로젝트에서나 사용 가능**하므로 PROJECT_ID 상태와 무관하게 항상 후보로 유지한다. FIELD_SP의 `Story Points` / `Story point estimate` 같은 글로벌 기본 필드는 이 규칙으로 통과한다.

#### 0-1-PROJECT — PROJECT_KEY + BOARD_ID 수집

config 파일이 없거나 PROJECT_KEY가 `YOUR_`로 시작하는 경우, 또는 0-0c에서 "프로젝트 키 / 보드 ID" 항목을 선택한 경우 진입한다.

1. AskUserQuestion: "Jira 프로젝트 키를 입력하세요. (예: TCI, MYPROJ)"
2. `jira_get_agile_boards`를 호출하여 해당 프로젝트의 보드 목록을 조회한다:
   - **결과 1개**: 자동 확정. "보드 `{보드명}`을 사용합니다." 안내 후 BOARD_ID 저장.
   - **결과 2~4개**: AskUserQuestion으로 보드 선택. 선택된 보드의 ID를 BOARD_ID로 저장.
   - **결과 5개 이상**: AskUserQuestion으로 "보드 이름 일부를 입력하세요."를 수집하고, 포함하는 보드만 필터 후 위 분기를 재적용.
   - **결과 0개**: "해당 프로젝트의 보드를 찾을 수 없습니다. 프로젝트 키를 확인하세요." 출력 후 중단.

#### 0-1-SP — FIELD_SP 슬롯 매칭

이 서브루틴을 단독 호출하는 경우 `jira_search_fields` 결과를 1회 조회하여 캐시한다. 같은 fallback 내에서 여러 슬롯을 순차 호출할 때는 앞서 조회한 결과를 재사용한다.

**0-0c 재진입 또는 0-0a 재지정 진입 시 표시**: 현재 값이 있으면 `현재 값: {customfield_xxxxx}. 재수집하시겠습니까? [예/아니오]`를 먼저 묻고, "아니오"면 해당 슬롯을 건드리지 않고 종료. 이 질문은 생략할 수 없다.

`jira_search_fields` 결과에서 `customfield_`로 시작하는 필드를 대상으로 아래 키워드 매칭을 적용한다(대소문자 무시).

**슬롯별 키워드 매칭 기준**

| 슬롯 | 키워드 |
|------|--------|
| FIELD_SP (스토리 포인트) | story point, storypoint, 스토리 포인트, 스토리포인트, SP |
| FIELD_AC (AC/완료조건) | acceptance, acceptance criteria, AC, 완료 조건, 완료조건, 인수 기준, 인수기준 |
| FIELD_EV (증거) | evidence, proof, 증거, 근거 |

FIELD_SP 슬롯 매핑:
- **매칭 1개**: "스토리 포인트 필드로 `{필드명} ({필드 키})`을 사용하시겠습니까?" 확인 후 사용.
- **매칭 0개 / 2개 이상**: AskUserQuestion 선택 UI 제시 (상위 3개 후보 + `사용 안 함` + `직접 입력`).
  - `사용 안 함` 선택: 해당 슬롯을 `(none)`으로 설정 → 관련 수집 Step 스킵.
  - `직접 입력` 선택: AskUserQuestion으로 필드 키 직접 입력.

#### 0-1-AC — FIELD_AC 슬롯 매칭

이 서브루틴을 단독 호출하는 경우 `jira_search_fields` 결과를 1회 조회하여 캐시한다. 같은 fallback 내에서 여러 슬롯을 순차 호출할 때는 앞서 조회한 결과를 재사용한다.

**0-0c 재진입 또는 0-0a 재지정 진입 시 표시**: 현재 값이 있으면 `현재 값: {customfield_xxxxx}. 재수집하시겠습니까? [예/아니오]`를 먼저 묻고, "아니오"면 해당 슬롯을 건드리지 않고 종료. 이 질문은 생략할 수 없다.

0-1-SP의 슬롯별 키워드 매칭 기준 표에서 FIELD_AC 행을 동일 규칙으로 적용하여 슬롯을 매핑한다. 매칭 수 분기(1개/0개·2개 이상)와 `사용 안 함`/`직접 입력` 처리도 0-1-SP와 동일하다.

#### 0-1-EV — FIELD_EV 슬롯 매칭

이 서브루틴을 단독 호출하는 경우 `jira_search_fields` 결과를 1회 조회하여 캐시한다. 같은 fallback 내에서 여러 슬롯을 순차 호출할 때는 앞서 조회한 결과를 재사용한다.

**0-0c 재진입 또는 0-0a 재지정 진입 시 표시**: 현재 값이 있으면 `현재 값: {customfield_xxxxx}. 재수집하시겠습니까? [예/아니오]`를 먼저 묻고, "아니오"면 해당 슬롯을 건드리지 않고 종료. 이 질문은 생략할 수 없다.

0-1-SP의 슬롯별 키워드 매칭 기준 표에서 FIELD_EV 행을 동일 규칙으로 적용하여 슬롯을 매핑한다. 매칭 수 분기(1개/0개·2개 이상)와 `사용 안 함`/`직접 입력` 처리도 0-1-SP와 동일하다.

config 로드 모드에서는 본문 스킬이 생성 직전에 `0-0a` 서브루틴을 호출하여 customfield 키 존재 여부를 1회 검증한다. 보드 재조회는 수행하지 않는다.

#### 0-1-SAVE — 신규 `{{CONFIG_PATH}}` 저장

**실행 조건**: 0-1 fallback 경로(config 미설정 → 0-1-PROJECT/SP/AC/EV 전체 순차 실행)로 진입한 경우에만 실행한다. 0-0c / 0-0a 재지정으로 슬롯별 단독 호출된 경우는 **건너뛴다**(해당 경로는 0-0c의 갱신 로직이 저장 담당). 0-2 오버라이드는 1회성이므로 저장하지 않는다.

절차:

1. `{{CONFIG_PATH}}`의 상위 디렉토리가 없으면 생성한다(예: `~/.agents/`가 없으면 `mkdir -p ~/.agents`).
2. 아래 포맷으로 파일 전체를 Write한다. `jira-create-setup.body.md`의 저장 포맷과 완전 동일해야 한다.

   ```
   프로젝트 키: {PROJECT_KEY}
   보드 ID: {BOARD_ID}
   스토리 포인트 필드: {FIELD_SP}
   AC 필드: {FIELD_AC}
   증거 필드: {FIELD_EV}
   기본 우선순위: (none)
   기본 증거 형태: (none)
   Slack 사용자 ID: (none)
   ```

   (`(none)` 문자열 그대로 기록. SLACK_ID는 인라인 fallback 규정에 따라 `(none)` 고정.)
3. 저장 완료 후 안내 출력: `"{{CONFIG_PATH}}에 설정을 저장했습니다. 다음 실행부터는 0-0 config 로드 경로로 진입합니다."`
4. 본문 `## Step 1`로 진입.

### 0-2. $ARGUMENTS 우선 적용

**0-2 경로는 1회성 오버라이드이므로 0-0b/0-0c/0-0a를 모두 건너뛰고 바로 보드·필드 재탐색으로 진입한다.**

**진입 조건 (AND — 모두 충족해야 함)**:
1. `$ARGUMENTS`에 알파벳 2~10자로 이루어진 단어가 포함되어 있다.
2. 그 단어가 작업 설명이 아닌 프로젝트 키로 판단된다.
3. **그 프로젝트 키가 config의 PROJECT_KEY와 서로 다른 값이다.** 같은 값이면 오버라이드 의미가 없으므로 0-2 경로로 진입하지 않는다 — 이 경우는 일반 경로로 `0-0b`부터 진행한다.

세 조건이 모두 충족되면 config의 PROJECT_KEY보다 우선 적용한다.

**오버라이드 시 보드/필드 재탐색 (필수)**: PROJECT_KEY를 오버라이드하는 경우, 기존 config의 BOARD_ID·FIELD_SP·FIELD_AC·FIELD_EV는 더 이상 유효하지 않다. 다음과 같이 처리한다:

1. "프로젝트 키 `{X}`로 1회성 생성합니다. 보드/필드를 재탐색합니다." 안내를 출력.
2. BOARD_ID·FIELD_SP·FIELD_AC·FIELD_EV를 무효화하고, 0-1-PROJECT → 0-1-SP → 0-1-AC → 0-1-EV 서브루틴을 순차 강제 실행한다.
3. SLACK_ID는 개인 알림이므로 그대로 유지한다.
4. 재탐색된 값은 `{{CONFIG_PATH}}`에 저장하지 않는다 (1회성).
