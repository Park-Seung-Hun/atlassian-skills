## Step 0 — 공통 설정

> **[필수] 실행 순서 정책**
> - 본 스킬을 호출하면 **반드시 이 `## Step 0`부터 순차 실행**한다.
> - 사용자 입력에 SDD 파일 경로·프로젝트 키·요약 등 작업 대상 정보가 포함되어 있어도 **Step 0를 건너뛰지 않는다**. 작업 대상 정보는 Step 1 이후에서 소비한다.
> - 처리 순서: `0-0 config 로드` → (로드 성공 시) `0-0b 현재 설정 확인` → 사용자 응답에 따라 `0-0c 항목별 수정`(루프) 또는 곧바로 본문 `## Step 1` 진입. config 로드 실패 시엔 `0-1` 전체 fallback 후 `## Step 1`.
> - `$ARGUMENTS`에 **config의 PROJECT_KEY와 다른** 프로젝트 키가 포함된 경우만 예외: `0-2` 경로로 진입하며 `0-0b/0-0c`를 스킵한다. **같은 프로젝트 키**가 포함된 경우(예: config PROJECT_KEY=JST인데 사용자가 "JST에"로 재언급한 경우)는 오버라이드가 아니므로 일반 경로로 0-0b부터 진행한다 (0-2 섹션 진입 조건 참조).
> - **Step 1 진입 조건**: `0-0b` 선택 완료 또는 `0-1` fallback 완료 또는 `0-2` 재탐색 완료.
> - **0-0a 자동 customfield 키 검증**은 Step 0 진입 단계에서 실행하지 않는다. 본문 스킬이 **Jira 생성 직전**(배치: Step 4 "이대로 생성" 확정 직후, 단건: Step 5 확인 직후)에 명시적으로 호출하는 **지연 실행 서브루틴**이다.

> **실행 환경 전제**
> - `mcp-atlassian`, `notion`, `slack` MCP 서버가 등록돼 있어야 한다(스킬에 따라 사용하는 서버는 일부만 해당).
> - 본문에서 `mcp__<server>__<tool>` 풀네임으로 표기된 도구는 Claude Code/Codex 양 환경에서 동일하게 모델에 노출된다.
> - 본문에 "AskUserQuestion"이라고 적힌 부분은 구조화 질문 도구를 의미한다. 해당 도구가 없는 환경(예: Codex)에서는 동일 의미의 자연어 질문으로 대체하되, 선택지/검증 조건은 본문에 명시된 그대로 유지한다.
> - 설정 파일 경로 `{{CONFIG_PATH}}`는 두 환경이 워크스페이스 루트 기준으로 공유한다.

### 커스텀 필드 설정

커스텀 필드 ID는 0-0(config 로드) 또는 0-1(config 없을 때의 인라인 수집)에서 확정된다.
이하 지시에서 `{FIELD_SP}` / `{FIELD_AC}` / `{FIELD_EV}`는 확정된 필드 키를 의미한다.
config 로드 경로인 경우 본문 스킬이 Jira 생성 직전에 `0-0a` 서브루틴을 호출하여 customfield 키의 존재 여부를 1회 검증한다.
`(none)`으로 설정된 슬롯은 해당 수집 Step 전체를 스킵한다.

> **중요**: `description` 필드에 AC/증거를 포함하면 Jira가 자동으로 파싱하여 섹션이 잘리거나 커스텀 필드에 잘못 매핑된다.
> description = 목적 + 범위만 작성하고, AC와 증거는 반드시 커스텀 필드로 분리하여 전달한다.

### 0-0. config 로드

`{{CONFIG_PATH}}` 파일을 읽어 아래 값을 로드한다.

- **PROJECT_KEY**: `프로젝트 키` 항목
- **BOARD_ID**: `보드 ID` 항목
- **FIELD_SP**: `스토리 포인트 필드` 항목 (`(none)`이면 스토리 포인트 수집 Step 스킵)
- **FIELD_AC**: `AC 필드` 항목 (`(none)`이면 AC 수집 Step 스킵)
- **FIELD_EV**: `증거 필드` 항목 (`(none)`이면 증거 수집 Step 스킵)
- **DEFAULT_PRIORITY**: `기본 우선순위` 항목 (없거나 `(none)`이면 스킬별 fallback 사용)
- **DEFAULT_EVIDENCE**: `기본 증거 형태` 항목 (없거나 `(none)`이면 스킬별 fallback 사용)
- **SLACK_ID**: `Slack 사용자 ID` 항목 (`(none)`이면 Slack 알림 Step 스킵)

로드 후 **PROJECT_ID 확보 + PROJECT_KEY_MAP 캐시**(세션 캐시용) — `jira_get_all_projects()`를 호출하여 다음을 한 번에 추출한다:
- 응답 배열에서 `key == {PROJECT_KEY}` 인 엔트리의 `id` → `PROJECT_ID`
- 응답 배열 전체를 순회하여 `id → key` 매핑 → `PROJECT_KEY_MAP` (0-1 슬롯별 후보 출처 라벨 표시에 사용. 별도 API 호출 없이 같은 응답에서 추출)

추출에 성공하면 세션에 캐시하고 이후 본 스킬 실행 동안 재조회하지 않는다. 0-0a / 0-1 슬롯별 서브루틴은 캐시값을 그대로 사용한다.

실패 시(API 오류, 매칭 엔트리 없음 등) 사용자에게 다음 메시지를 출력하고 **스킬을 종료**한다. 자동 재시도하지 않는다:

> "프로젝트 목록 조회에 실패했거나 `{PROJECT_KEY}`와 일치하는 프로젝트를 찾지 못했습니다. 잠시 후 스킬을 다시 실행하거나, PROJECT_KEY를 확인해 주세요."

### 0-0b. 현재 설정 확인

> **[차단 지점]** 이 절은 Step 1로 진입하기 전 **반드시 사용자 응답을 받고 종료**한다. config 로드가 성공한 경우 요약 출력과 선택지 질문을 생략할 수 없다. "바로 본문을 실행하는 게 사용자 의도로 보인다"는 판단으로 이 절을 건너뛰는 것을 금지한다.

**실행 조건**: 0-0에서 config 로드가 성공하고, 0-2 $ARGUMENTS 오버라이드가 발동되지 않은 경우에만 실행. config 파일이 없거나 PROJECT_KEY가 `YOUR_`로 시작하면 0-0b를 건너뛰고 0-1로 진입. 0-2 경로도 0-0b/0-0c/0-0a 전체를 건너뜀.

로드된 config 값을 아래 형식으로 요약 출력한다.

```
현재 설정 ({{CONFIG_PATH}}):
- 프로젝트 키       : {PROJECT_KEY}
- 보드 ID           : {BOARD_ID}
- 스토리 포인트 필드: {FIELD_SP}
- AC 필드           : {FIELD_AC}
- 증거 필드         : {FIELD_EV}
- 기본 우선순위     : {DEFAULT_PRIORITY}
- 기본 증거 형태    : {DEFAULT_EVIDENCE}
- Slack 사용자 ID   : {SLACK_ID}
```

AskUserQuestion (단일 선택):
> "위 설정으로 진행할까요, 항목을 수정할까요?"
- **그대로 진행** → 본문 `## Step 1` 진입 (customfield probe는 생성 직전에 수행)
- **항목 수정** → 0-0c 진입

### 0-0c. 항목별 수정

AskUserQuestion (**복수 선택**):
> "수정할 항목을 선택하세요."

선택지:
- 프로젝트 키 / 보드 ID (함께 수정) → **0-1-PROJECT** 재실행
- 스토리 포인트 필드 (FIELD_SP) → **0-1-SP** 재실행
- AC 필드 (FIELD_AC) → **0-1-AC** 재실행
- 증거 필드 (FIELD_EV) → **0-1-EV** 재실행
- Slack 사용자 ID → **0-1-SLACK** 재실행 (표시명 자동 변환 또는 직접 ID 입력. `(none)` 선택 시 알림 비활성화)
- 기본 우선순위 / 기본 증거 형태 → **0-1-DEFAULTS** 재실행 (개별 항목만 다시 묻고 싶더라도 두 슬롯을 함께 재수집한다. 이전 값 유지를 원하면 해당 슬롯에서 동일 값을 다시 선택)

선택된 항목을 순차적으로 수집한 뒤 `{{CONFIG_PATH}}` 갱신.

**config 갱신 로직**:
1. `{{CONFIG_PATH}}`를 Read로 로드 (전체 내용).
2. 저장 포맷은 `라벨: 값` 라인 단위. 각 항목의 라벨은 아래와 같이 고정(이 라벨이 바뀌면 0-1-SAVE의 신규 작성 흐름과 sprint 계열 스킬의 incremental 갱신 사이 호환성이 깨진다):
   - `프로젝트 키:`, `보드 ID:`, `스토리 포인트 필드:`, `AC 필드:`, `증거 필드:`, `기본 우선순위:`, `기본 증거 형태:`, `Slack 사용자 ID:`
3. 수정된 슬롯에 해당하는 **라인만** 새 값으로 교체. 다른 라인과 파일 내 다른 섹션은 건드리지 않는다.
4. Write로 전체 파일을 덮어쓴다.
5. "`{{CONFIG_PATH}}`를 갱신했습니다." 안내 출력.

완료 후 **0-0b로 복귀**하여 갱신된 값을 재확인한다 (사용자가 다시 "수정"을 선택하면 추가 수정 가능).

### 0-0a. 자동 customfield 키 검증 (지연 실행 서브루틴)

> **[중요] 호출 규약**
> - 이 서브루틴은 Step 0 진입 단계에서 **자동 실행되지 않는다**. 본문 스킬이 Jira 생성 직전에 명시적으로 호출한다 (배치: Step 4 "이대로 생성" 확정 직후 · 단건: Step 5 확인 직후, Step 6 첫 호출 이전).
> - 호출 조건: config 로드 경로(0-0 성공)로 진입한 세션에 한함. `0-1` fallback 완료 세션과 `0-2` 오버라이드 세션은 호출하지 않는다 (수집 중 사용자 명시 선택으로 이미 확정됨 / 1회성 오버라이드).
> - **입력**: 0-0에서 확정·캐시된 `PROJECT_KEY`, `PROJECT_ID`, `FIELD_SP` / `FIELD_AC` / `FIELD_EV` 값.
> - **출력**: 아래 3가지 중 하나.
>   - `통과` — 모든 슬롯 검증 통과. config/세션 변경 없음.
>   - `재지정` — 실패 슬롯을 재지정·비활성화 경로로 처리 완료. config 또는 세션 필드 맵이 변경됨. 호출 측 본문은 **생성 전 반드시 사용자 재확인**을 수행한다 (아래 "재지정 반환 후 호출 측 재확인" 참조).
>   - `중단` — 사용자가 중단 선택. 호출 측 본문은 생성을 수행하지 않고 스킬 종료.
> - **세션 내 멱등**: 한 세션에서 `통과`로 반환된 뒤에는 본문 스킬이 재호출하더라도 실제 probe(`jira_search_fields`)를 재실행하지 않는다. 세션 플래그 `customfield_probe_passed = true`로 기록하고 즉시 `통과` 반환. 재지정으로 config가 갱신된 경우는 플래그가 해제(false)된 상태에서 새로 probe한다.

로드된 customfield 키가 현재 Jira 인스턴스에서 실제 조회 가능한지 1회 검증한다.

**probe**:
- config의 FIELD_SP / FIELD_AC / FIELD_EV 각 customfield 키를 `jira_search_fields`의 `keyword` 인자로 **슬롯별 호출**한다 (예: `jira_search_fields({keyword: "customfield_10026", limit: 5})`). `(none)`으로 설정된 슬롯은 호출 대상에서 제외하므로 최대 3회 호출. 모든 슬롯이 `(none)`이면 probe 자체를 수행하지 않고 즉시 `통과` 반환.
  - 인스턴스 customfield 수가 많은 환경에서 `keyword=""` + `limit:200` 식의 전역 호출은 응답 절단으로 글로벌 필드가 누락될 수 있어 사용하지 않는다.
  - 응답이 0건이거나 응답에 `id == {슬롯 키}`인 항목이 없으면 해당 슬롯은 **실패 (미존재)**로 판정한다.
  - 동일 키로 재호출하거나 limit을 키워가며 재시도하는 것을 금지한다.
- config의 FIELD_SP / FIELD_AC / FIELD_EV 각 값에 대해 아래 순서로 판정한다:
  1. `id` 필드가 일치하는 항목을 찾는다. 없으면 **실패 (미존재)**.
  2. 해당 항목의 `scope` 필드는 출처 정보 용도로만 사용하며, 검증 통과/실패 판정에 쓰지 않는다 (Jira의 customfield 공유 메커니즘상 `scope.project.id`가 다른 프로젝트더라도 현재 프로젝트에 공유 사용 중일 수 있다). 1번 단계에서 `id`만 일치하면 **통과**.

> 이 검증은 "config가 가리키는 customfield가 인스턴스에 실제 존재하는지"만 확인한다. 사용자가 명시적으로 선택한 customfield는 출처가 어느 프로젝트든 인정한다.

**실패 감지 시 분기 (슬롯별 순차 처리)**:

어떤 슬롯이 위 판정에서 실패하면 해당 슬롯에 대해 아래를 수행한다. 실패 사유에 따라 안내 문구를 달리한다. **안내 문구에는 실패 사실만 담고, 대체 후보값은 절대 함께 노출하지 않는다** (후보는 0-1-{슬롯} 재진입 UI에서만 제시).

```
# 미존재 — 안내 예시 (이 문구 외 정보 추가 금지)
FIELD_AC(`customfield_99999`)가 jira_search_fields 결과에서 확인되지 않습니다.
```

AskUserQuestion (단일 선택):
> "어떻게 처리할까요?"
- **재지정** → 해당 슬롯의 0-1 서브섹션(`0-1-AC` 등) 재진입 → 새 값 확정 → 0-0c의 config 갱신 로직으로 `{{CONFIG_PATH}}` 저장 → 다음 실패 슬롯으로 이동.
- **비활성화** → 세션 내에서만 해당 슬롯을 `(none)`으로 처리. `{{CONFIG_PATH}}`는 **변경하지 않는다**(의도적). 다음 실행 시 동일 오류가 재발할 수 있음을 안내.
- **중단** → 스킬 즉시 종료. 서브루틴은 `중단`을 반환하고 호출 측 본문은 생성을 수행하지 않는다.

> **[필수] 재지정 경로에서 지켜야 할 3가지 (어느 하나라도 빠뜨리지 말 것)**
>
> **(1) 0-1 서브섹션 재진입 강제**
> 재지정 선택 직후 **0-1-SP / 0-1-AC / 0-1-EV 중 해당 슬롯 서브섹션을 반드시 호출**한다. "현재 값: {customfield_xxxxx}. 재수집하시겠습니까? [예/아니오]" 질문을 먼저 표시하고(이 질문은 0-0c 재진입뿐 아니라 **0-0a 재지정 진입 시에도 동일하게 표시**한다), "아니오"면 해당 슬롯 비활성화 또는 다음 슬롯으로 진행. "예"면 키워드 매칭 → 출처 라벨 변환 → 후보 제시 → "**매칭 1개인 경우에도 확인 질문 → 예/아니오 응답 수신**" 또는 "**매칭 0개/2개 이상인 경우 후보 선택 UI**"를 반드시 거친 뒤에만 새 값을 확정한다.
>
> **(2) config 저장 도구 제한**
> 확정된 값을 `{{CONFIG_PATH}}`에 반영할 때 사용 가능한 도구는 **`Read`와 `Write` 단 두 가지뿐**이다. 아래 도구·방식을 사용한 config 치환을 **전부 금지**한다:
> - `Bash` 도구를 통한 `sed`, `perl`, `awk`, `gsed`, `python -c`, `ruby -pi`, `cat > ...`, `echo "..." > ...`, `tee`, 리다이렉션(`>`/`>>`) 등.
> - `Edit` 도구의 in-place 부분 치환.
>
> 저장 절차는 정확히: `Read`로 전체 파일 로드 → 메모리에서 해당 라인만 교체 → `Write`로 전체 파일 덮어쓰기. 이 순서·도구 조합을 벗어난 저장은 수행하지 않는다. `Write` 실패 시 `Bash` 치환 도구로 우회하지 말고 사용자에게 저장 실패 사실을 알리고 중단한다.
>
> **(3) 실패 안내에 후보값 동반 노출 금지**
> "미존재" 안내 문구에는 **실패 사실만** 담는다. "현재 JST에서 쓰이는 AC 후보는 …" 같은 대체 후보를 같이 출력하거나, 실패 안내와 재지정 UI를 같은 턴에 합쳐 "후보 X로 바꿀까요?" 류의 단일 질문으로 축약하는 것을 금지. 대체 후보는 반드시 0-1-{슬롯} 재진입 UI에서만 제시한다.

모든 실패 슬롯 처리가 끝나거나 검증이 바로 통과하면 `customfield_probe_passed = true`를 세팅하고 **호출 측 본문으로 복귀**한다 (Step 1로 점프하지 않는다 — 호출 지점에 따라 Step 5 또는 Step 6으로 진행). 이때 config/세션 필드 맵이 변경됐다면 `재지정`을, 변경이 없었다면 `통과`를 반환 값으로 넘긴다.

> **[필수] 재지정 반환 후 호출 측 재확인**
> 서브루틴이 `재지정`을 반환한 경우, 호출 측 본문은 **즉시 Jira 생성 호출을 수행하지 않는다**. 아래 질문을 먼저 출력하고 응답을 받은 뒤에만 다음 단계로 진입한다.
> - AskUserQuestion(단일 선택): "설정이 변경됐습니다 ({변경 슬롯 요약, 예: `AC 필드: customfield_99999 → customfield_12881`}). 이 설정으로 생성을 이어갈까요?"
>   - **이어서 생성** → Step 5(배치) 또는 Step 6(단건) 진행.
>   - **취소** → 생성 없이 종료.
> 이전 (C) "생성" 응답을 근거로 사용자 재확인을 생략하고 바로 `jira_create_issue` / `jira_batch_create_issues` 등을 호출하는 것을 금지한다. config 상태가 달라졌으므로 사용자 동의를 새로 받아야 한다.

### 0-1. 인라인 수집 서브루틴

아래 여섯 개 서브섹션은 독립적으로 호출 가능하다. config 미설정 fallback에서는 0-1-PROJECT → 0-1-SP → 0-1-AC → 0-1-EV → 0-1-SLACK → 0-1-DEFAULTS 순차 호출 후 반드시 **0-1-SAVE**로 `{{CONFIG_PATH}}`를 신규 작성한다. 0-0c/0-0a 재지정 분기는 해당 슬롯 서브섹션만 호출하며, 저장은 0-0c의 갱신 로직이 담당한다.

> **[필수] 6단계 분리 호출 강제 (단축 금지)**
>
> - 0-1 fallback에 진입하면 **6개 서브섹션을 반드시 별도 단계로 분리** 실행한다(PROJECT → SP → AC → EV → SLACK → DEFAULTS). 한 화면에 "보드 / SP / AC / EV / 우선순위 / 증거 형태 / Slack" 등 여러 슬롯 후보를 묶어 한 번에 묻고 사용자 응답을 합쳐 받는 단축 흐름을 **금지**한다. 묶음 미리보기로 단계를 점프하면 슬롯별 확인 질문(0-1-SP/AC/EV의 매칭 1개 확인, 0-1-SLACK 사용 여부, 0-1-DEFAULTS 두 슬롯)이 누락된다.
> - 0-1-DEFAULTS와 0-1-SLACK은 **모델이 자체 추정으로 채워서 건너뛰는 것을 금지**한다. 두 서브섹션 모두 사용자 응답 없이 진입을 마칠 수 없다. 사용자가 "Slack 비활성화"라고 선언했더라도 0-1-SLACK 1단계 질문(사용 여부)은 출력하고 응답을 받아 분기한다 — 그 응답이 곧 "사용 안 함" 선택지에 매핑된다.
> - 첫 슬롯에서 사용자가 가치를 미리 알려줬어도 그 슬롯의 확인 질문은 건너뛰지 않는다.

> **[필수] 사용자 확인 강제 (역추정 자동 채택 금지)**
>
> - 사용자 프롬프트에 프로젝트 키·필드 값이 이미 포함되어 있어도 슬롯별 확인 질문을 건너뛰지 않는다. "사용자가 이미 값을 줬다"는 판단으로 자동 확정하는 것을 금지한다.
> - 기존 이슈를 `jira_get_issue(fields="*all")` 등으로 조회해 customfield를 **역추정한 결과를 사용자 확인 없이 바로 채택하는 행위를 금지**한다. 역추정 자체는 허용되나 결과는 반드시 "매칭 1개" 분기의 확인 질문 또는 "매칭 0개/2개 이상" 분기의 선택 UI를 거쳐 사용자가 명시적으로 확정해야 한다. **"JST-XXX에서 customfield_YYYYY가 실제 사용 중이니 이 값을 채택합니다" 류의 자동 채택 + 묶음 미리보기 + 단일 확인 질문 단축 흐름을 명시적으로 금지**한다(세 슬롯 각각 별도 확인 질문).
> - "매칭 1개"인 경우에도 `"스토리 포인트 필드로 {필드명} ({필드 키})을 사용하시겠습니까?"` 형태의 확인 질문을 **반드시 출력**하고 예/아니오 응답을 받은 뒤에만 확정한다.

> **[필수] customfield 후보 출처 라벨링 (0-1-SP / 0-1-AC / 0-1-EV 공통)**: `jira_search_fields` 결과에서 키워드 매칭으로 받은 customfield 후보들을 **필터링하지 않고 모두 후보 풀로 사용**한다. 대신 각 후보의 `scope` 정보를 사용자가 판단할 수 있도록 출처 라벨로 변환해 표시한다.
>
> Jira는 한 customfield가 여러 프로젝트에서 공유 사용될 수 있는 메커니즘을 제공한다(Team-managed 공유, field configuration scheme 등). `jira_search_fields`의 `scope.project.id`는 그 customfield가 *원래 정의된* 프로젝트만 알려주고 *공유 사용 여부*는 응답에 포함되지 않으므로, scope 기반 필터링은 false negative를 만들 수 있다. 따라서 필터하지 않고 출처를 사용자에게 그대로 보여 판단을 위임한다.
>
> **출처 라벨 규칙** (각 후보를 사용자에게 제시할 때 customfield 키 옆에 한 줄로 표시):
> - `scope` 키 **없음** → `[글로벌]`
> - `scope.type == "PROJECT"` 이고 `scope.project.id == PROJECT_ID` → `[{현재 PROJECT_KEY} 정의]`
> - `scope.type == "PROJECT"` 이고 `scope.project.id != PROJECT_ID` → `[{KEY} 정의]` (KEY는 0-0에서 캐시한 `PROJECT_KEY_MAP`에서 `scope.project.id`로 조회)
> - 위에서 PROJECT_KEY_MAP 조회 실패(매핑 누락) → `[id={scope.project.id}]` fallback
>
> **후보 풀 0건 처리**: 키워드 매칭 자체가 0건이면 사용자에게 아래 안내를 출력하고 `사용 안 함` / `직접 입력`만 제시한다.
> > "키워드 매칭에서 `{슬롯}` 후보를 찾지 못했습니다. 필드 key를 직접 입력하거나 사용 안 함을 선택하세요."

#### 0-1-PROJECT — PROJECT_KEY + BOARD_ID 수집

config 파일이 없거나 PROJECT_KEY가 `YOUR_`로 시작하는 경우, 또는 0-0c에서 "프로젝트 키 / 보드 ID" 항목을 선택한 경우 진입한다.

1. AskUserQuestion: "Jira 프로젝트 키를 입력하세요. (예: TCI, MYPROJ)"
2. `jira_get_agile_boards`를 호출하여 해당 프로젝트의 보드 목록을 조회한다:
   - **결과 1개**: 자동 확정. "보드 `{보드명}`을 사용합니다." 안내 후 BOARD_ID 저장.
   - **결과 2~4개**: AskUserQuestion으로 보드 선택. 선택된 보드의 ID를 BOARD_ID로 저장.
   - **결과 5개 이상**: AskUserQuestion으로 "보드 이름 일부를 입력하세요."를 수집하고, 포함하는 보드만 필터 후 위 분기를 재적용.
   - **결과 0개**: "해당 프로젝트의 보드를 찾을 수 없습니다. 프로젝트 키를 확인하세요." 출력 후 중단.

#### 0-1-SP — FIELD_SP 슬롯 매칭

이 서브루틴은 `jira_search_fields`를 FIELD_SP 슬롯의 키워드로 호출한다. 아래 매칭 표의 키워드 중 하나를 `keyword` 인자로 사용한다 (예: `jira_search_fields({keyword: "story point", limit: 50})`). 같은 fallback 내에서 AC/EV 슬롯으로 진행할 때는 각각 새 키워드로 다시 호출한다 (응답 캐시 공유 없음).

> **[필수] 슬롯별 키워드 호출**: SP/AC/EV는 슬롯마다 독립적인 `jira_search_fields` 호출을 사용한다. `keyword=""` + `limit:200` 식의 전역 호출은 인스턴스 customfield 수가 많을 때 응답 절단으로 글로벌 SP 필드(`Story Points`, `Story point estimate` 등)가 누락될 수 있어 사용하지 않는다. 동일 슬롯 안에서 응답이 잘려 보인다고 limit을 키워가며 재시도하는 것은 금지(`limit: 50` 정도면 슬롯별 fuzzy 매칭은 충분).

**응답 스키마**: `jira_search_fields` 응답 배열의 각 항목은 `id`, `name`, `schema.type`을 포함하며, project 전용 customfield인 경우 `scope.type == "PROJECT"`와 `scope.project.id`도 함께 들어 있다. `scope` 키 자체가 없으면 글로벌 필드.

**응답 처리 순서** (받은 직후):
1. **출처 라벨 변환** — 아래 "[필수] customfield 후보 출처 라벨링" 절의 규칙에 따라 각 후보의 `scope`를 `[글로벌]` / `[{KEY} 정의]` / `[id={n}]` 형태의 라벨로 변환한다. **필터링하지 않는다.** 키워드 매칭 결과를 모두 후보 풀로 유지한다.
2. **매칭 수 분기** — 후보 풀 항목 수(키워드 매칭 응답 그대로)를 기준으로 아래 슬롯 매핑을 적용한다.

**0-0c 재진입 또는 0-0a 재지정 진입 시 표시**: 현재 값이 있으면 `현재 값: {customfield_xxxxx}. 재수집하시겠습니까? [예/아니오]`를 먼저 묻고, "아니오"면 해당 슬롯을 건드리지 않고 종료. 이 질문은 생략할 수 없다.

**슬롯별 키워드 매칭 기준** (이 표의 키워드 중 하나를 `jira_search_fields(keyword=...)` 인자로 사용. 슬롯당 1회 호출.)

| 슬롯 | 키워드 |
|------|--------|
| FIELD_SP (스토리 포인트) | story point, storypoint, 스토리 포인트, 스토리포인트, SP |
| FIELD_AC (AC/완료조건) | acceptance, acceptance criteria, AC, 완료 조건, 완료조건, 인수 기준, 인수기준 |
| FIELD_EV (증거) | evidence, proof, 증거, 근거 |

FIELD_SP 슬롯 매핑 (키워드 매칭 응답 후보 수 기준):
- **매칭 1개**: "스토리 포인트 필드로 `{필드명} ({필드 키}) {출처 라벨}`을 사용하시겠습니까?" 확인 후 사용. 예: `스토리 포인트 필드로 Story Points (customfield_10028) [글로벌]을 사용하시겠습니까?`
- **매칭 0개 / 2개 이상**: AskUserQuestion 선택 UI 제시 (상위 3개 후보 + `사용 안 함` + `직접 입력`). 각 후보는 `{필드명} ({필드 키}) {출처 라벨}` 형식으로 표시. 예: `Story Points (customfield_10028) [글로벌]`, `Story Point (customfield_10513) [QA 정의]`.
  - `사용 안 함` 선택: 해당 슬롯을 `(none)`으로 설정 → 관련 수집 Step 스킵.
  - `직접 입력` 선택: AskUserQuestion으로 필드 키 직접 입력.

#### 0-1-AC — FIELD_AC 슬롯 매칭

이 서브루틴은 `jira_search_fields`를 FIELD_AC 슬롯의 키워드로 호출한다 (예: `keyword: "acceptance"`, `keyword: "인수 기준"`). 응답을 받은 직후 0-1-SP와 동일하게 ① 출처 라벨 변환 → ② 매칭 수 분기 순서로 처리한다.

**0-0c 재진입 또는 0-0a 재지정 진입 시 표시**: 현재 값이 있으면 `현재 값: {customfield_xxxxx}. 재수집하시겠습니까? [예/아니오]`를 먼저 묻고, "아니오"면 해당 슬롯을 건드리지 않고 종료. 이 질문은 생략할 수 없다.

0-1-SP의 슬롯별 키워드 매칭 기준 표에서 FIELD_AC 행의 키워드를 사용하고, 매칭 수 분기(1개/0개·2개 이상)와 `사용 안 함`/`직접 입력` 처리는 0-1-SP와 동일하게 적용한다.

#### 0-1-EV — FIELD_EV 슬롯 매칭

이 서브루틴은 `jira_search_fields`를 FIELD_EV 슬롯의 키워드로 호출한다 (예: `keyword: "evidence"`, `keyword: "증거"`). 응답을 받은 직후 0-1-SP와 동일하게 ① 출처 라벨 변환 → ② 매칭 수 분기 순서로 처리한다.

**0-0c 재진입 또는 0-0a 재지정 진입 시 표시**: 현재 값이 있으면 `현재 값: {customfield_xxxxx}. 재수집하시겠습니까? [예/아니오]`를 먼저 묻고, "아니오"면 해당 슬롯을 건드리지 않고 종료. 이 질문은 생략할 수 없다.

0-1-SP의 슬롯별 키워드 매칭 기준 표에서 FIELD_EV 행의 키워드를 사용하고, 매칭 수 분기(1개/0개·2개 이상)와 `사용 안 함`/`직접 입력` 처리는 0-1-SP와 동일하게 적용한다.

config 로드 모드에서는 본문 스킬이 생성 직전에 `0-0a` 서브루틴을 호출하여 customfield 키 존재 여부를 1회 검증한다. 보드 재조회는 수행하지 않는다.

#### 0-1-SLACK — Slack 사용자 ID 수집

**0-0c 재진입 시 표시**: 현재 SLACK_ID 값이 있으면 `현재 값: {SLACK_ID}. 재수집하시겠습니까? [예/아니오]`를 먼저 묻고, "아니오"면 해당 슬롯을 건드리지 않고 종료.

1. AskUserQuestion (단일 선택): "Slack 알림을 사용하시겠습니까?"
   - **사용 안 함** → SLACK_ID = `(none)`으로 설정 후 종료. 이후 본문 스킬의 Slack 알림 Step은 자동 스킵된다.
   - **사용** → 2번으로 진행.

2. AskUserQuestion: "Slack 표시 이름(display name)을 입력하세요."

3. `slack_get_users`로 전체 사용자 목록을 1회 조회한 뒤 입력된 표시명과 일치하는 `id`를 찾는다.
   - **일치 1건**: "Slack 사용자 `{display_name} ({id})`을 사용하시겠습니까?" 확인 후 SLACK_ID 저장.
   - **일치 0건 / 2건 이상**: 아래 안내 출력 후 직접 입력 받기.
     > "Slack 표시명으로 사용자를 자동 식별하지 못했습니다. Slack 앱 → 본인 프로필 클릭 → '더보기' → '멤버 ID 복사'로 확인할 수 있습니다."
     - AskUserQuestion: "Slack member ID를 직접 입력하세요. (예: U01ABCDEFGH, 비활성화하려면 `(none)` 입력)"
     - 빈 값이거나 `(none)` 입력 시 SLACK_ID = `(none)`.

#### 0-1-DEFAULTS — 기본 우선순위 + 기본 증거 형태 수집

> **[필수] 자동 추정 채택 금지**: 모델이 `Medium` / `PR 링크` 등 추천값을 추정해 사용자 응답 없이 채택·저장하는 것을 **금지**한다. 두 슬롯 모두 AskUserQuestion으로 사용자 응답을 받아야 진입을 마친다. 사용자가 "그대로 진행"이라고 답한 경우에도 그 응답은 0-0b의 "현재 설정 확인" 응답으로만 해석되며, 0-1-DEFAULTS 진입 시점에는 별도의 두 단계 질문이 필요하다.

**0-0c 재진입 시 표시**: 현재 값이 있으면 `현재 값: 기본 우선순위={DEFAULT_PRIORITY}, 기본 증거 형태={DEFAULT_EVIDENCE}. 재수집하시겠습니까? [예/아니오]`를 먼저 묻고, "아니오"면 해당 슬롯을 건드리지 않고 종료.

##### 1. 기본 우선순위

`jira-batch-create` 자동 보강·`jira-create` 미리보기 추천에 사용한다. 미설정 시 본문 스킬이 fallback `Medium`을 사용한다.

AskUserQuestion (단일 선택):
> "Jira 이슈 생성 시 기본 우선순위를 선택하세요."
- Highest / High / **Medium (추천)** / Low / 설정 안 함

"설정 안 함" 선택 시 DEFAULT_PRIORITY = `(none)`.

##### 2. 기본 증거 형태 (PBI/Sub-task)

`jira-batch-create` 자동 보강에 사용한다. 미설정 시 본문 스킬이 fallback `PR 링크`(PBI/Sub-task) / `배포 URL`(Epic)을 사용한다.

AskUserQuestion (단일 선택):
> "PBI/Sub-task의 기본 증거 형태를 선택하세요."
- **PR 링크 (추천)** / Confluence 문서 링크 / 스크린샷·로그 / 설정 안 함

"설정 안 함" 선택 시 DEFAULT_EVIDENCE = `(none)`.

#### 0-1-SAVE — 신규 `{{CONFIG_PATH}}` 저장

**실행 조건**: 0-1 fallback 경로(config 미설정 → 0-1-PROJECT/SP/AC/EV/SLACK/DEFAULTS 전체 순차 실행)로 진입한 경우에만 실행한다. 0-0c / 0-0a 재지정으로 슬롯별 단독 호출된 경우는 **건너뛴다**(해당 경로는 0-0c의 갱신 로직이 저장 담당). 0-2 오버라이드는 1회성이므로 저장하지 않는다.

> **[필수] 저장 도구 제한** (0-0c·0-0a와 동일 규약): 사용 가능한 도구는 **`Read`와 `Write` 단 두 가지뿐**이다. `Bash`를 통한 `cat > ... <<EOF`, `echo "..." > ...`, `tee`, 리다이렉션(`>`/`>>`), `sed`/`perl`/`awk`로 config 파일을 작성하는 것을 전부 금지. `mkdir -p`만은 디렉토리 생성을 위해 허용한다. **`Write` 도구가 환경 sandbox 등으로 실패한 경우, `Bash` heredoc 등으로 우회하지 말고 사용자에게 저장 실패 사실을 알리고 중단**한다("config 파일을 저장하지 못했습니다 ({사유}). 세션 값으로 진행할지, 권한을 부여하고 재시도할지 알려주세요." 안내).

절차:

1. `{{CONFIG_PATH}}`의 상위 디렉토리가 없으면 생성한다(예: `~/.agents/`가 없으면 `mkdir -p ~/.agents`). 디렉토리 생성에만 `Bash`를 사용하며, 파일 내용 작성에는 사용하지 않는다.
2. 아래 포맷으로 **`Write` 도구로 파일 전체를 작성**한다. 라벨은 0-0c 갱신 로직과 호환을 유지하기 위해 고정.

   ```
   # 개인 설정 (Personal Config)
   # ⚠️ 이 파일은 개인 정보를 포함합니다. 절대 git에 커밋하지 마세요.

   ## Jira
   프로젝트 키: {PROJECT_KEY}
   보드 ID: {BOARD_ID}
   스토리 포인트 필드: {FIELD_SP}
   AC 필드: {FIELD_AC}
   증거 필드: {FIELD_EV}
   기본 우선순위: {DEFAULT_PRIORITY}
   기본 증거 형태: {DEFAULT_EVIDENCE}

   ## 알림
   Slack 사용자 ID: {SLACK_ID}   # (none)이면 Slack 알림 비활성

   ## Notion
   스프린트 DB Data Source: YOUR_SPRINT_DB_DATASOURCE_ID
   스프린트 플래너 DB Data Source: YOUR_PLANNER_DB_DATASOURCE_ID
   동기화 로그 DB Data Source: YOUR_SYNC_LOG_DB_DATASOURCE_ID
   업무 템플릿 ID: YOUR_TEMPLATE_PAGE_ID
   ```

   (`(none)` 문자열은 사용자가 0-1-SLACK / 0-1-DEFAULTS에서 "사용 안 함" / "설정 안 함"을 선택한 경우 그대로 기록한다. Notion 섹션은 jira-create 계열에서 사용하지 않으며 `/sprint-setup` 실행 시 채워진다.)
3. 저장 완료 후 안내 출력: `"{{CONFIG_PATH}}에 설정을 저장했습니다. 다음 실행부터는 0-0 config 로드 경로로 진입합니다."`
4. 본문 `## Step 1`로 진입.

### 0-2. $ARGUMENTS 우선 적용

**0-2 경로는 1회성 오버라이드이므로 0-0b/0-0c/0-0a를 모두 건너뛰고 바로 보드·필드 재탐색으로 진입한다.**

**진입 조건 (AND — 모두 충족해야 함)**:
1. `$ARGUMENTS`에 알파벳 2~10자로 이루어진 단어가 포함되어 있다.
2. 그 단어가 작업 설명이 아닌 프로젝트 키로 판단된다.
3. **그 프로젝트 키가 config의 PROJECT_KEY와 서로 다른 값이다.** 같은 값이면 오버라이드 의미가 없으므로 0-2 경로로 진입하지 않는다 — 이 경우는 일반 경로로 `0-0b`부터 진행한다.

세 조건이 모두 충족되면 config의 PROJECT_KEY보다 우선 적용한다.

**오버라이드 시 보드/필드 재탐색 (필수)**: PROJECT_KEY를 오버라이드하는 경우, 기존 config의 BOARD_ID·FIELD_SP·FIELD_AC·FIELD_EV는 더 이상 유효하지 않다. 다음과 같이 처리한다:

1. "프로젝트 키 `{X}`로 1회성 생성합니다. 보드/필드를 재탐색합니다." 안내를 출력.
2. BOARD_ID·FIELD_SP·FIELD_AC·FIELD_EV를 무효화하고, 0-1-PROJECT → 0-1-SP → 0-1-AC → 0-1-EV 서브루틴을 순차 강제 실행한다.
3. SLACK_ID는 개인 알림이므로 그대로 유지한다.
4. 재탐색된 값은 `{{CONFIG_PATH}}`에 저장하지 않는다 (1회성).
